# C/C++ Crash Analyzer

## Quick start

1. **Get the inputs.** Ask for (or find) whichever of these exist — don't block on all of them:
   - The crash log itself (pasted or attached file, e.g. `@crashlog.txt`)
   - The binary that crashed (built **with debug symbols**, `-g`, ideally unstripped)
   - The source tree / repo (to read the actual lines once located)
   - If ASAN: the raw sanitizer output already contains a symbolized trace, so binary access is optional but source access is still needed.

2. **Identify the log format** (see table below) — this determines how you extract addresses/frames.

3. **Symbolize.** Resolve raw addresses to `file:line` using `scripts/symbolize.sh <binary> <addr> [addr...]` (wraps `addr2line`), or by loading the core in gdb (`gdb <binary> <core>` then `bt full`). ASAN/gdb-with-symbols output is often already symbolized — skip straight to reading source.

4. **Read the actual source** at the crash frame (top of stack = where it faulted; also check the caller for how bad data got there).

5. **Classify the pattern** — null deref, use-after-free, out-of-bounds, stack overflow, dangling/wild pointer, etc. See [REFERENCE.md](REFERENCE.md) for signatures and how to distinguish them.

6. **Report** using the format in "Output format" below.

## Log format cheat sheet

| Signal in the log | Format | How to symbolize |
|---|---|---|
| `#0  0x0000... in func_name (...)` lines | GDB backtrace | Usually already has `at file.c:123`; if only addresses, run `symbolize.sh` |
| `Program terminated with signal SIGSEGV` / `Core was generated by` | GDB core-dump report | Load with `gdb <binary> <core>`, run `bt full` |
| `... segfault at ... ip 0000... sp 0000... error 4 in binary[...]` | Linux kernel dmesg/klog line | Extract `ip` (instruction pointer) offset, use `symbolize.sh` against the binary at that offset (subtract the load base if ASLR — see REFERENCE.md) |
| `ERROR: AddressSanitizer: SEGV on unknown address ...` with `#0 0x... in func file.c:12:5` | ASAN report | Already symbolized; read directly. Note the "READ/WRITE of size N" and shadow byte legend — tells you the exact bad access |
| `*** stack smashing detected ***` | Stack protector (fortify) | Different from plain SEGV — canary corruption, look for buffer writes in the flagged function |

If the format doesn't match any of these, say so explicitly and ask the user what tool produced it rather than guessing.

## Workflow checklist

- [ ] Confirm binary/symbols availability; if stripped, tell the user symbolization will be limited to addresses/offsets and suggest rebuilding with `-g -O0`
- [ ] Extract the full backtrace, not just frame #0 — the call chain often explains *why* a bad pointer arrived, not just where it was dereferenced
- [ ] Symbolize every frame you can
- [ ] Open the actual source file(s) at the reported lines
- [ ] Check variable/pointer lifetime: where was it allocated, freed, initialized, bounds-checked?
- [ ] Classify against [REFERENCE.md](REFERENCE.md) patterns
- [ ] If ASAN output present, use its shadow-memory legend to confirm the classification (heap-use-after-free, heap-buffer-overflow, stack-buffer-overflow, etc.) instead of guessing
- [ ] Propose a minimal, targeted fix (null check, bounds check, ownership/lifetime fix, correct free, increase stack, etc.) — don't over-refactor

## Output format

Report back with:

1. **Crash site** — `file:line` (function), and the one-line reason (e.g. "null pointer dereference reading `node->value`")
2. **Call chain** — condensed backtrace showing how execution got there
3. **Root cause classification** — one of the categories in REFERENCE.md, with the specific evidence from the log that supports it
4. **Why it happened** — the concrete code condition (e.g. "`ptr` is freed on line 40 then used again on line 55 without being set to NULL")
5. **Suggested fix** — a small, concrete code change or guard, shown as a diff/snippet against the real source
6. **Confidence & gaps** — if symbols were missing, source wasn't available, or the log was ambiguous, say so plainly instead of overstating certainty
